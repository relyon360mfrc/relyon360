## #5 — Requisições de folga (Comunicação) não chegavam ao Supabase

### Sintoma
- Instrutor (Bismarck) submetia requisição de folga.
- Para o admin/planejador, a requisição **nunca aparecia** — nem em real-time, nem após F5, nem em outro dispositivo.
- Mesmo comportamento acontecia com a Cobertura Diária (`relyon_activities`).

### Diagnóstico
Query direta no Supabase mostrou que a chave `relyon_requests` **não existia em `app_state`** — mesmo após várias submissões. A causa: a política RLS de INSERT em `app_state` tinha uma allowlist de chaves explícita, e essa allowlist **não incluía** `relyon_requests` nem `relyon_activities`. O `usePersisted` chama `upsert`, que vira INSERT na primeira gravação, e o INSERT era bloqueado silenciosamente pela RLS — o erro era emitido mas não chamava atenção do usuário.

Allowlist antiga (8 chaves):
```
relyon_schedules, relyon_trainings, relyon_areas, relyon_instructors,
relyon_users, relyon_absences, relyon_locals, relyon_holidays
```

Faltavam: `relyon_requests` e `relyon_activities` (e mais tarde `relyon_class_tombstones` foi adicionado pelo fix do #3).

### Fix
Migração SQL `rls_app_state_allow_requests_and_activities`:

```sql
DROP POLICY IF EXISTS app_state_insert ON public.app_state;

CREATE POLICY app_state_insert
  ON public.app_state
  FOR INSERT
  TO anon
  WITH CHECK (
    key = ANY (ARRAY[
      'relyon_schedules'::text,
      'relyon_trainings'::text,
      'relyon_areas'::text,
      'relyon_instructors'::text,
      'relyon_users'::text,
      'relyon_absences'::text,
      'relyon_locals'::text,
      'relyon_holidays'::text,
      'relyon_activities'::text,
      'relyon_requests'::text
      -- 'relyon_class_tombstones' adicionado na migração seguinte (fix #3)
    ])
  );
```

Nenhuma mudança de código no app foi necessária — o `usePersisted` já tentava o upsert correto, só que a RLS estava barrando.

### Como verificar se voltou a quebrar
Console do navegador no app:
```js
await sb.from('app_state').select('key').eq('key', 'relyon_requests');
```
Se retornar `data: []` mesmo depois de uma requisição ter sido submetida, a RLS voltou a estar quebrada. Conferir política via SQL:
```sql
SELECT with_check FROM pg_policies
WHERE tablename='app_state' AND cmd='INSERT';
```
Deve conter `relyon_requests` e `relyon_activities`.

### Rollback
Indesejável (volta a perder dados). Se mesmo assim for necessário, reverter as duas migrações via SQL recriando a política antiga.

## #4 — Botão "Forçar sincronização" sem feedback visual

### Sintoma
Botão no painel inferior direito ficava cinza fixo. Usuário clicava e nada acontecia — sem sinal de progresso, sem confirmação, sem erro. Impossível saber se algo havia rodado.

### Diagnóstico
No componente `SaveMonitor` (`js/app.js`):

```js
const flushNow = () => { if (window.__outboxFlush) window.__outboxFlush(); };
```

`flushNow` era **fire-and-forget**: não await, não mostra estado, não captura resultado. E quando `stats.total === 0` (caso comum, sem fila), `_outboxFlush` retorna silenciosamente sem fazer nada. Combinado com o estilo cinza fixo do botão, o clique era 100% imperceptível.

### Fix
Em `js/app.js`, `SaveMonitor`:

1. Dois novos estados:
   ```js
   const [flushing,    setFlushing]    = React.useState(false);
   const [flushResult, setFlushResult] = React.useState(null);
   ```

2. `flushNow` agora é async, captura `before`/`after` da outbox e expõe o resultado:
   ```js
   const flushNow = React.useCallback(async () => {
     setFlushing(true);
     setFlushResult(null);
     const before = window.__outboxStats();
     try {
       await window.__outboxFlush();
     } catch (e) { /* mostra erro */ }
     const after = window.__outboxStats();
     // decide entre 'empty' (nada pendente), 'done' (N resolvidas),
     // 'partial' (N ainda em retry/RLS)
     setFlushing(false);
     setTimeout(() => setFlushResult(null), 6000); // auto-some em 6s
   }, [refreshStats]);
   ```

3. Botão muda visualmente:
   - `flushing===true` → fundo escuro, texto "Sincronizando…", `disabled`
   - `flushing===false` → fundo laranja/cinza, cursor pointer

4. Caixa de resultado acima do botão mostra:
   - "✓ N operação(ões) sincronizada(s)" (verde)
   - "✓ Nada pendente — banco já está sincronizado" (cinza)
   - "N pendente(s) — N com erro de permissão (RLS)" (vermelho)

### Como verificar se voltou a quebrar
Abrir o painel inferior direito (clicar no chip de status). Clicar no botão "Forçar sincronização" / "Sincronizar agora". Deve sempre aparecer uma mensagem de resultado em até 1-2 segundos.

### Rollback
Reverter o trecho do `SaveMonitor` em `js/app.js` para o estado anterior ao commit `490792f`. Procurar pelos dois estados `flushing` e `flushResult` para identificar o que adicionar/remover.

## #3 — Turmas excluídas ressuscitando

### Sintoma
Usuário deletava uma turma. Pouco tempo depois (geralmente após F5, troca de dispositivo, ou simplesmente "do nada"), a turma reaparecia na tela. Já tinha sido tentado corrigir 4 vezes; tombstone local (commit `d8c9eb0`) reduziu mas não eliminou.

### Diagnóstico — dois cenários distintos

**Cenário A — single-device, F5 logo após apagar:**
1. `deleteClass` → `setSchedules` filtra o state (turma some) → tombstone marcado em memória/LS → `_deleteSchedulesByClassId` dispara `_persistQueue` (async)
2. Usuário aperta F5 **antes do DELETE async terminar no Supabase**
3. Boot fetch `sb.from('relyon_schedules').select('*')` traz as rows **porque o DELETE ainda não rodou**
4. Reconciliação no `useSchedules` filtrava apenas `pendingLocal` pelo tombstone, mas **não filtrava `all`** (o que vem do Supabase)
5. `merged = [...all, ...pendingLocal]` incluía as rows fantasmas → LS contaminado → turma volta

**Cenário B — multi-device:**
1. Aba A apaga turma → tombstone marcado em A (per-device, em LS local)
2. Aba B estava offline ou com Realtime fora de sync → nunca recebeu o evento DELETE
3. Aba B volta online e faz F5
4. `prev` (LS de B) ainda tem a turma. `all` (Supabase) não tem (foi apagada por A).
5. `_isClassDeleted` em B retorna `false` (tombstone existe só em A) → row vai pra `pendingLocal` → B re-insere no Supabase → turma volta para todo mundo

### Fix — em duas frentes

**Frente 1 (Cenário A) — `js/config.js`, dentro de `useSchedules`:**

Filtrar `all` também pelo tombstone E re-disparar DELETE para ghosts:

```js
const ghosts = all.filter(s => _isClassDeleted(s.classId));
const cleanAll = all.filter(s => !_isClassDeleted(s.classId));
if (ghosts.length > 0) {
  const ghostClassIds = [...new Set(ghosts.map(s => s.classId))];
  console.warn(`[useSchedules] ${ghosts.length} ghost row(s) (classId tombstoned) ainda no Supabase. Re-deletando ${ghostClassIds.length} classId(s).`);
  ghostClassIds.forEach(cid => _deleteSchedulesByClassId(cid));
}
const sbIds = new Set(cleanAll.map(s => String(s.id)));
const pendingLocal = prev.filter(s => !sbIds.has(String(s.id)) && !_isClassDeleted(s.classId));
const merged = pendingLocal.length > 0 ? [...cleanAll, ...pendingLocal] : cleanAll;
```

**Frente 2 (Cenário B) — tombstone global via `app_state`:**

1. Migração SQL `rls_app_state_allow_class_tombstones` adicionou `'relyon_class_tombstones'` à allowlist de INSERT do `app_state`.

2. `_DB_KEYS` em `config.js` ganhou a chave:
   ```js
   const _DB_KEYS = [..., 'relyon_class_tombstones'];
   ```

3. `_markClassDeleted` agora espelha no Supabase via fila serial:
   ```js
   let _tombstoneSyncQueue = Promise.resolve();
   const _syncTombstoneToSupabase = (map) => {
     _tombstoneSyncQueue = _tombstoneSyncQueue.then(async () => {
       await sb.from('app_state').upsert(
         { key: 'relyon_class_tombstones', value: map },
         { onConflict: 'key' }
       );
     });
   };
   ```

4. Nova função `_hydrateTombstonesFromInitialData` (exposta como `window.__hydrateTombstones`) que popula `_deletedClassIdsMemory` no boot a partir do que veio do Supabase. Respeita TTL global de **7 dias** (vs 4h do TTL local).

5. AppLoader em `js/app.js` chama `__hydrateTombstones()` **antes do useSchedules rodar**:
   ```js
   if (typeof window.__hydrateTombstones === 'function') {
     window.__hydrateTombstones();
   }
   ```

### Estrutura final do tombstone (3 camadas)
1. **Memória** (`_deletedClassIdsMemory` Set) — guard imediato, mais rápido
2. **localStorage** (`rl360_deleted_classes`, TTL 4h) — sobrevive F5 single-device
3. **Supabase** (`app_state[key='relyon_class_tombstones']`, TTL 7d) — global, fecha o gap multi-device

### Como verificar se voltou a quebrar
1. **Single-device:** crie uma turma teste, apague, dê F5 imediatamente — não pode voltar.
2. **Multi-device:** apague em um dispositivo, em outro dispositivo (sem Realtime) dê F5 — não pode voltar.
3. Console do navegador:
   ```js
   // ver tombstones locais
   JSON.parse(localStorage.getItem('rl360_deleted_classes'))
   // ver tombstones globais
   await sb.from('app_state').select('value').eq('key','relyon_class_tombstones');
   ```

### Rollback
Mais complexo. Reverter o commit `490792f` em `js/config.js` (a seção TOMBSTONE inteira e a reconciliação do `useSchedules`) e em `js/app.js` (a chamada `__hydrateTombstones` no AppLoader). A migração RLS pode ficar — chave a mais na allowlist não atrapalha.
